## 技巧 1：并行处理 - 一个人活成一支队伍

### 💡 啥意思？
想象一下，如果你能同时出现在 3 个不同的地方做 3 件事，效率会提升多少？

这就是并行处理的威力。

### 🎯 具体怎么做？

#### 第一步：创建多个工作区
```bash
# 在你的项目目录下创建多个分支的工作区
git worktree add ../project-feature-a feature-a
git worktree add ../project-feature-b feature-b
git worktree add ../project-bugfix bugfix
```

这会在你的电脑上创建 3 个独立的文件夹，每个都是一个完整的项目副本。

#### 第二步：分别打开不同的 Claude 会话
- 会话 1：处理 feature-a 的开发
- 会话 2：处理 feature-b 的开发
- 会话 3：修复 bug

**这就像你有 3 个分身同时在干活！**

#### 第三步（可选）：设置快捷切换
```bash
# 在你的 ~/.zshrc 或 ~/.bashrc 中添加别名
alias za='cd ../project-feature-a'
alias zb='cd ../project-feature-b'
alias zc='cd ../project-bugfix'
```

现在你只需要敲 `za`、`zb`、`zc` 就能在不同任务间快速切换。

### 📊 对比一下

| 普通人 | 顶级玩家 |
|--------|----------|
| 一个一个任务串行做 | 3-5 个任务同时做 |
| 等待一个任务完成才能开始下一个 | 利用等待时间处理其他任务 |
| 1 小时完成 3 个任务 | 1 小时完成 9 个任务 |

### 🌟 实战案例

假设你要：
1. 开发一个新功能（需要 30 分钟）
2. 修复一个 bug（需要 15 分钟）
3. 重构一段旧代码（需要 20 分钟）

**普通做法**：总共需要 65 分钟

**并行做法**：
- 同时开启 3 个会话
- 3 个任务并行推进
- 总共只需要 30 分钟左右（最长的那个任务的时间）

## 技巧 2：计划模式先行 - 先画图纸再盖房子

### 💡 啥意思？

很多人用 AI 就像带实习生：
- "帮我写个函数"
- "这段代码有 bug，帮我修"
- "把这里改一下"

这种方式的问题：你是在"砌砖"，一块一块往上堆。

顶级玩家的做法：先让 AI 当"建筑师"，画好完整图纸，再让它当"工人"按图施工。

### 🎯 具体怎么做？

#### 第一步：开两个 Claude 会话

**会话 A（建筑师）**：专门负责做计划
```
我要在这个项目中添加用户认证功能，包括：
- 用户注册
- 用户登录
- 密码找回
- 记住登录状态

请先给我一个详细的实现计划，包括：
1. 需要创建哪些文件
2. 需要修改哪些现有文件
3. 实现的先后顺序
4. 可能遇到的问题和解决方案
```

**会话 B（资深工程师）**：专门负责审查计划
```
这是我另一个会话生成的实现计划：
[粘贴会话 A 的计划]

请以资深工程师的角度审查这个计划：
1. 是否有遗漏的地方？
2. 是否有更好的实现方式？
3. 是否存在潜在风险？
4. 给出你的改进建议
```

#### 第二步：合并建议，执行计划
```
好的，根据你的审查意见，修改后的计划如下：
[整合后的最终计划]

现在请按照这个计划，一步一步实现。每完成一步，告诉我进度。
```

#### 第三步：遇到问题立即回到计划模式
如果实现过程中出了问题，不要硬着头皮往下修！

```
遇到了一个问题：[描述问题]

让我先停下来，回到计划模式，重新审视这个方案是否需要调整。
```

### 📊 对比一下

| 普通人 | 顶级玩家 |
|--------|----------|
| 直接让 AI 写代码 | 先让 AI 制定详细计划 |
| 写到一半发现方向错了 | 提前发现潜在问题 |
| 推倒重来浪费时间 | 方案正确，一气呵成 |

### 🌟 实战案例

假设你要添加"购物车"功能：

**❌ 错误做法**：
```
Claude，帮我写一个购物车功能
```
→ Claude 可能写了一堆代码，结果发现不符合你的需求

**✅ 正确做法**：
```
请先给我一个购物车功能的实现计划：
1. 数据结构设计
2. API 设计
3. 前端组件设计
4. 实现步骤
```
→ 先把方案想清楚，再动手写代码

## 技巧 3：用心维护 CLAUDE.md - 让 AI 越用越聪明

### 💡 啥意思？

CLAUDE.md 就像是你给 AI 写的"使用说明书"。

每次 AI 犯错后，你纠正它，然后把错误和正确做法记录到 CLAUDE.md 里。

慢慢地，AI 就学会你的偏好和规则，不会再犯同样的错误。

### 🎯 具体怎么做？

#### 第一步：在项目根目录创建 CLAUDE.md
```markdown
# Claude 项目规范

## 代码风格
- 使用 4 空格缩进
- 变量命名用驼峰式
- 组件名用帕斯卡式

## 禁止做的事情
- 不要使用 jQuery
- 不要直接操作 DOM
- 不要写内联样式

## 项目特定规则
- 所有 API 请求都要经过 api.js 这个统一入口
- 用户数据必须存在 localStorage 中，key 是 'userData'
- 错误处理必须显示在页面的右上角

## 常见错误
❌ 错误：直接用 fetch 调用 API
✅ 正确：使用 api.js 封装的 request 方法
```

#### 第二步：每次 AI 犯错，就更新 CLAUDE.md

**场景**：Claude 写了一段代码，直接用了 `fetch`

```
[粘贴 Claude 写的代码]

这段代码违反了项目规范，应该使用 api.js。

请修改代码，并且更新你的 CLAUDE.md，记录这条规则，下次不要再犯。
```

#### 第三步（进阶）：建立知识库目录

```
project/
├── CLAUDE.md          # 主规范文件
└── .claude/
    ├── auth-flow.md   # 认证流程说明
    ├── api-guide.md   # API 使用指南
    └── db-schema.md   # 数据库结构
```

在 CLAUDE.md 中引用：
```markdown
## 参考文档
- 认证流程：.claude/auth-flow.md
- API 使用：.claude/api-guide.md
- 数据库结构：.claude/db-schema.md
```

### 📊 效果对比

| 使用 CLAUDE.md 前 | 使用 CLAUDE.md 后 |
|------------------|------------------|
| 每次都要提醒 AI 不要用 jQuery | AI 自动遵守规则 |
| AI 经常犯同样的错误 | 犯错率明显下降 |
| 需要反复纠正 | 一次纠正，终身受益 |

### 🌟 实战案例

假设你的项目不能用任何第三方库：

**第一次 AI 写了**：
```javascript
import _ from 'lodash';  // ❌ 禁止
```

你纠正它，并要求更新 CLAUDE.md。

**下次 AI 就会知道**：
```javascript
// 不用 lodash，自己写工具函数
const isEmpty = (obj) => Object.keys(obj).length === 0;
```

## 技巧 4：创建可复用技能 - 把重复工作变成一键操作

### 💡 啥意思？

如果你发现某个操作每天都要做超过一次，那就把它变成一个"技能"或"命令"。

就像游戏里的"快捷键"，一键释放一整套连招。

### 🎯 具体怎么做？

#### 示例 1：创建"技术债务清理"命令

**场景**：每次工作结束前，你想检查是否有重复代码

**第一步**：创建一个 `.claude/skills/techdebt.md` 文件
```markdown
---
name: techdebt
description: 检查并清理技术债务
---

请帮我检查当前项目：
1. 查找重复的代码片段
2. 查找未使用的 import
3. 查找可以简化的复杂函数
4. 给出改进建议
```

**第二步**：在 Claude 中使用
```
/techdebt
```

Claude 就会按照你定义的步骤，自动检查项目。

#### 示例 2：创建"周报生成"命令

**场景**：每周五要生成工作周报，需要汇总 Slack、GitHub、邮件等信息

**第一步**：创建 `.claude/skills/weekly-report.md`
```markdown
---
name: weekly-report
description: 生成周报
---

请帮我生成本周工作周报：

1. 从我的 GitHub 仓库获取过去 7 天的 commits
2. 查找我在 Slack 上的重要发言
3. 查找邮件中的关键事项
4. 汇总成以下格式：
   - 本周完成的功能
   - 本周修复的 bug
   - 下周计划
   - 遇到的问题
```

**第二步**：每周五只需输入
```
/weekly-report
```

#### 示例 3：创建"代码审查"命令

```markdown
---
name: review
description: 代码审查
---

请审查以下代码：
[当前打开的文件或选中的代码]

审查要点：
1. 代码逻辑是否正确
2. 是否有潜在 bug
3. 是否符合项目规范
4. 性能是否有优化空间
5. 安全性是否有问题
```

### 📊 效果对比

| 每次手动操作 | 使用技能命令 |
|------------|-------------|
| "帮我检查重复代码" | `/techdebt` |
| "帮我生成本周周报" | `/weekly-report` |
| "帮我审查这段代码" | `/review` |
| 需要重复输入详细要求 | 一个命令搞定 |

### 🌟 核心价值

> 把一次性的努力，变成可以无限复用的资产

这就是"复利"的威力！

## 技巧 5：让 Claude 自己修 bug - 学会放权

### 💡 啥意思？

**传统做法**：
- 看到 bug → 自己分析 → 自己定位 → 自己修 → 再让 AI 帮忙优化

**新做法**：
- 看到 bug → 直接扔给 AI → AI 自己分析、定位、修复 → 你来验收

### 🎯 具体怎么做？

#### 场景 1：修复 CI 失败的测试

**简单粗暴版**：
```
Claude，去把 CI 里挂掉的测试修好。别问我怎么做，你自己搞定，修好了告诉我。
```

**版本控制版**：
```
GitHub 的 CI 测试失败了，这是失败的链接：
[CI 链接]

请帮我：
1. 分析失败原因
2. 修复代码
3. 本地测试通过后，提交 PR
```

#### 场景 2：从 Slack 讨论

**第一步**：复制 Slack 上的 bug 讨论串

**第二步**：粘贴给 Claude
```
这是我们团队在 Slack 上讨论的一个 bug：

[粘贴 Slack 讨论记录]

请根据这些信息，定位并修复这个 bug。
```

#### 场景 3：复杂的分布式系统问题

```
这是 Docker 的日志：
[粘贴日志]

这是相关的代码文件：
[粘贴代码]

请分析这个分布式系统的问题，并给出解决方案。
```

### 📊 对比一下

| ❌ 传统做法 | ✅ 新做法 |
|-----------|----------|
| 自己花 30 分钟分析 | 让 AI 花 5 分钟分析 |
| 自己定位 bug 位置 | AI 自己定位 |
| 自己写修复代码 | AI 自己修复 |
| 自己测试 | AI 自己测试 |
| **你累得半死** | **你只需验收** |

### 🌟 背后的理念

**信任 + 授权** = 把 AI 当成真正的合作伙伴

你不再是个"操作员"，而是个"管理者"。

## 技巧 6：提升提示词水平 - 把 AI 变成你的私人教练

### 💡 啥意思？

不是让 AI "帮你做事"，而是让 AI "逼你做得更好"。

### 🎯 具体怎么做？

#### 场景 1：让 AI 当你的"严格代码审查员"

```
Claude，现在请你扮演一个严格的代码审查专家。

我的要求：
1. 仔细审查我的代码改动
2. 找出所有潜在问题
3. 不准放水，不准说"看起来不错"
4. 在我通过你的所有测试之前，不准让我创建 PR

开始审查：
[粘贴代码]
```

#### 场景 2：让 AI "证明"方案可行

```
Claude，我打算这样实现这个功能：
[你的方案]

请向我证明这个方案是可行的：
1. 分析可能的风险
2. 对比主分支和功能分支的行为差异
3. 找出边界情况
4. 如果有更好的方案，直接告诉我，不要委婉
```

#### 场景 3：拒绝平庸，追求卓越

```
Claude，刚才你给我的方案能 work，但不够优雅。

现在你已经了解了所有情况，把刚才那个方案扔掉。

给我一个：
- 更简洁的实现
- 更好的性能
- 更易读的代码

我要的是一个让我会"哇"的方案，而不是"能用就行"。
```

#### 场景 4：基础很重要 - 写详细的需求

**❌ 模糊需求**：
```
帮我写一个登录功能
```
→ AI 只能猜，结果往往不满意

**✅ 详细需求**：
```
帮我实现登录功能，具体要求：

## 功能需求
1. 用户输入邮箱和密码
2. 点击"登录"按钮
3. 验证成功后跳转到首页
4. 验证失败显示错误提示

## 技术要求
- 使用现有的 api.js
- Token 存在 localStorage
- 错误提示显示在右上角
- 登录按钮要有 loading 状态

## 设计稿
[附上设计稿截图]

## 参考代码
可以参考现有的注册功能：src/pages/Register.jsx
```

### 📊 对比一下

| 把 AI 当工具 | 把 AI 当教练 |
|------------|-------------|
| "帮我写代码" | "审查我的代码，别客气" |
| "这个能 work 吗？" | "证明给我看这个方案可行" |
| 满足于"能用就行" | 追求"优雅和卓越" |
| AI 给出平庸方案 | 要求 AI 给出最佳方案 |

### 🌟 核心转变

**从"AI 帮我干活"到"AI 帮我变强"**

## 技巧 7：优化物理交互 - 说话比打字快三倍

### 💡 啥意思？

我们说话的速度通常是打字的 3 倍。

用语音输入，你能在同样的时间内给 AI 提供更详细、更丰富的指令。

### 🎯 具体怎么做？

#### macOS 语音输入

**第一步**：开启语音输入
- 连按两下 `fn` 键
- 或者点击菜单栏的输入法图标，选择"语音"

**第二步**：对着电脑说话
```
嘿 Claude，我需要你帮我重构一下这个登录表单。

具体要求是这样的：
第一，把邮箱输入框和密码输入框分开成两个独立的组件
第二，添加一个"记住我"的复选框
第三，把登录按钮改成圆角的，颜色改成蓝色
第四，当用户点击登录的时候，要显示一个 loading 状态
第五，如果登录失败，要在右上角显示错误提示，而不是 alert
第六，密码输入框要有一个"显示密码"的小眼睛图标

参考一下注册页面的样式，保持一致。
```

**优势**：
- 同样的时间，你能说 300 字，只能打 100 字
- 语气和停顿能传达更多信息
- 不会因为"懒得打字"而省略细节

#### 终端优化：状态栏

在你的 `.zshrc` 或 `.bashrc` 中添加：

```bash
# 显示当前 git 分支和上下文用量
function git_branch() {
    echo "$(git symbolic-ref --short HEAD 2>/dev/null)"
}

export PROMPT='%F{blue}$(git_branch)%f %F{green}%~%f $ '
```

效果：
```
feature-login ~/project/user-auth $
```

一眼就能看到：
- 当前分支：feature-login
- 当前目录：user-auth

#### 终端优化：tmux 标签页

```bash
# 安装 tmux（如果没有）
brew install tmux

# 创建会话并命名标签页
tmux new-session -s work
tmux rename-window "auth"
tmux new-window -n "api"
tmux new-window -n "frontend"

# 给不同标签页设置不同颜色（在 ~/.tmux.conf 中）
set -g window-status-format "#[fg=colour%d]#I:#W"
```

效果：
```
[auth] [api] [frontend]
```

一眼就知道哪个窗口在干什么。

### 📊 效果对比

| 打字输入 | 语音输入 |
|---------|---------|
| "帮我改下登录" | "我要添加记住我功能，改成圆角按钮..." |
| 省略细节 | 包含所有细节 |
| AI 需要反复追问 | 一次说清楚，减少沟通成本 |
| 输入 100 字/分钟 | 输入 300 字/分钟 |

### 🌟 核心价值

**让输入速度跟上你的思考速度**

## 技巧 8：使用子智能体 - CEO 只做决策，不干杂活

### 💡 啥意思？

Claude 有一个功能叫"子智能体"（subagents）。

你可以把它理解为：AI 可以"召唤"更多 AI 帮忙。

### 🎯 具体怎么做？

#### 基础用法：增加算力

当任务特别复杂时，在请求后面加上：
```
[你的问题或任务]

use subagents
```

这会让 Claude 投入更多计算资源来解决问题。

#### 常见用法：分配子任务

```
我需要重构这个项目：

主任务：重构用户认证模块

请分别处理以下子任务，每个子任务用单独的子智能体：
1. 子任务 A：重构登录逻辑
2. 子任务 B：重构注册逻辑
3. 子任务 C：重构密码重置逻辑

主会话只负责汇总结果，不做具体实现。
```

**好处**：
- 每个子智能体专注于一个任务
- 主会话保持干净，不会因为上下文太长而混乱
- 就像 CEO 把任务分配给不同部门

#### 高级用法：自动化权限管理

```
当有权限请求时，自动路由给 Opus 4.5（一个更强大的 AI）：
1. 让 Opus 4.5 扫描是否有安全问题
2. 如果安全，自动批准
3. 如果有风险，拒绝并说明原因
```

这就像给公司配了一个"自动审批系统"。

### 📊 对比一下

| 不用子智能体 | 用子智能体 |
|------------|-----------|
| 主会话变得很混乱 | 主会话保持清晰 |
| 上下文太长，AI 遗忘前面内容 | 每个子智能体专注自己的任务 |
| 你要手动管理多个会话 | 自动分配，自动汇总 |
| 你是"打杂的" | 你是"CEO" |

### 🌟 核心理念

**专注主线，授权下放**

就像真正的 CEO：
- 不会亲自去修打印机
- 不会亲自去写代码
- 专注于决策和统筹

## 技巧 9：用 AI 做数据分析 - 6 个月没写过 SQL

### 💡 啥意思？

Boris（Claude Code 创造者）说：
> "我已经超过 6 个月没写过一行 SQL 代码了。"

他怎么查数据？
直接让 AI 查。

### 🎯 具体怎么做？

#### 第一步：准备工具

假设你用的是 BigQuery（Google 的数据库工具）：
```bash
# 安装 bq 命令行工具
gcloud components install bq

# 配置认证
gcloud auth login
```

#### 第二步：让 AI 执行查询

在 Claude 中：
```
我想查询上个月新注册用户的地域分布。

请用 bq 工具查询：
1. 统计每个省份的用户数
2. 按照用户数降序排列
3. 只显示用户数大于 100 的省份
```

Claude 会：
1. 自动生成 SQL 语句
2. 执行 bq 命令
3. 把结果展示给你

#### 第三步：进一步分析

```
从这个数据看，广东和北京的用户最多。

请进一步分析：
1. 这两个省份用户的活跃度如何？
2. 他们的付费率是多少？
3. 和其他省份相比有什么特点？
```

Claude 会继续查询，给你深入分析。

### 🌟 适用范围

不只是 BigQuery，任何有命令行工具、API 的数据库都可以：

**MySQL**：
```
请连接我们的 MySQL 数据库，查询...
```

**PostgreSQL**：
```
请用 psql 查询...
```

**MongoDB**：
```
请用 mongo 查询...
```

**Redis**：
```
请用 redis-cli 查询...
```

### 📊 效果对比

| 传统方式 | AI 方式 |
|---------|--------|
| 自己写 SQL：`SELECT * FROM...` | 告诉 AI：我想查询什么 |
| 忘记 SQL 语法，到处搜文档 | AI 自动生成正确的 SQL |
| 查询结果要自己分析 | AI 帮你分析数据 |
| 重复写类似的查询 | AI 记住你的查询模式 |
| **你是数据分析师** | **AI 是你的数据分析师** |

### 🌟 核心价值

**你不学 SQL，AI 就是你的 SQL 专家**

## 技巧 10：把 AI 当学习伙伴 - 像费曼技巧一样学习

### 💡 啥意思？

费曼学习法：最好的学习方式是"把知识教给别人"。

AI 可以扮演"学生"的角色，你向它解释，它来提问和纠正。

### 🎯 具体怎么做？

#### 第一步：启用"解释模式"

在 Claude 的 `/config` 中：
```
{
  "outputStyle": "explanatory"
}
```

这样，Claude 每次改动代码都会解释"为什么这样改"。

#### 第二步：让 AI 讲解代码

```
Claude，这段代码我看不懂：
[粘贴代码]

请生成一个 HTML 演示文稿，逐行讲解这段代码。
```

Claude 会生成一个可交互的 HTML，像幻灯片一样逐段讲解。

#### 第三步：让 AI 画结构图

```
Claude，我需要理解这个新协议。

请画一个 ASCII 结构图，展示：
1. 数据流向
2. 各个组件的关系
3. 关键的处理步骤
```

Claude 会画出类似这样的图：
```
Client → API Gateway → Auth Service
              ↓
         Rate Limiter
              ↓
         Business Logic
              ↓
         Database
```

#### 第四步：费曼学习法

```
Claude，我要向你解释"React Hook 的工作原理"，请找出我理解中的漏洞。

我的理解是：[你的解释]

请你：
1. 指出我理解错误的地方
2. 追问一些我可能没理解的问题
3. 填补我知识的空白
4. 把最终的正确理解保存下来
```

### 🌟 间隔重复学习

创建一个 `.claude/skills/spaced-repetition.md`：

```markdown
---
name: learn
description: 费曼学习法 - 向 AI 解释来巩固知识
---

请扮演我的学习伙伴：

1. 我向你解释一个概念
2. 你找出我理解中的漏洞
3. 你追问一些深度问题
4. 我们一起完善理解
5. 把最终的正确理解保存到 .claude/knowledge/

今天要学习的是：[用户输入的主题]
```

使用：
```
/learn React Hook
```

### 📊 效果对比

| 传统学习 | AI 辅助学习 |
|---------|-----------|
| 看文档，似懂非懂 | AI 逐行讲解代码 |
| 被动接受知识 | 主动向 AI 解释，发现漏洞 |
| 容易遗忘 | AI 帮你定期复习 |
| 孤立学习 | AI 建立知识网络 |

### 🌟 核心价值

**AI 不只是工具，更是你的私人导师和学习伙伴**

## 总结：从"使用 AI"到"与 AI 协作"

看完这 10 个技巧，你会发现一个核心差异：

| 普通用户 | 顶级玩家 |
|---------|----------|
| **把 AI 当工具** | **把 AI 当伙伴** |
| 问问题 → 得答案 | 多个会话并行推进 |
| 一个会话用到底 | 先计划再执行 |
| AI 犯错 → 自己修 | 维护 CLAUDE.md，让 AI 学习 |
| 每次重复输入 | 创建可复用技能 |
| 自己修 bug | 让 AI 自己修 bug |
| 满足于能用 | 让 AI 教自己做得更好 |
| 打字输入 | 语音输入，速度提升 3 倍 |
| 一个人干 | 分配子智能体，像 CEO |
| 自己写 SQL | 让 AI 查数据 |
| 用 AI 生成内容 | 把 AI 当学习伙伴 |

### 最后的话

**我们与顶级玩家的差距，不在于工具版本，而在于思维模式。**

他们没有把 AI 当成一个简单的问答机器或代码生成器，而是把它视为：
- ✅ 一个可以并行作战的伙伴
- ✅ 一个可以持续调教的系统
- ✅ 一个可以教学相长的老师
- ✅ 一个可以突破物理限制的效率放大器

**这或许才是拥抱 AI 时代的正确姿势。**
